## Qué se implementó en Fase 1

### Backend (FastAPI)
- `backend/app/main.py` — FastAPI app con CORS, routers, `/health`
- `backend/app/config.py` — pydantic-settings (DB URL, JWT secrets, CORS)
- `backend/app/database.py` — SQLAlchemy 2.0 engine + session
- `backend/app/security.py` — bcrypt password hashing, JWT encode/decode
- `backend/app/dependencies.py` — `get_current_user` con HTTPBearer

### Models (7 tablas)
- `users` — username, email, hashed_password, is_active, is_superuser
- `comisionistas` — nombre
- `tarifas` — comisionista_id, tipo (porcentaje/fijo_kg), valor
- `orden_items` — fecha, numero_orden, finca, producto, cantidad, unidad, precio_unitario, total, sector, estado (activo/liquidado/anulado)
- `asignaciones` — orden_item_id + comisionista_id (many-to-many)
- `liquidaciones` — nombre, mes, fecha_creacion
- `liquidacion_items` + `liquidacion_item_tarifas` — snapshot inmutable de órdenes al liquidar

### Routers
- `/api/v1/comisionistas` — CRUD + tarifas
- `/api/v1/ordenes` — CRUD, batch insert, asignar/desasignar comisionistas, asignar-global
- `/api/v1/liquidaciones` — crear con snapshot, detalle, eliminar, restaurar
- `/api/v1/reportes` — resumen, por-finca, por-producto, por-comisionista

### Infra
- `docker-compose.yml` — postgres:16-alpine + backend con hot reload
- `backend/Dockerfile` — Python 3.12 slim
- `backend/alembic/` — configurado para migraciones
- `backend/app/commands/seed_demo.py` — datos de demo (22 órdenes, 7 comisionistas, 3 liquidaciones)
- `backend/app/commands/create_superuser.py` — CLI para crear admin

### Diseño aprobado
- `docs/superpowers/specs/2026-05-22-fastapi-backend-migration-design.md` — spec completo
- `docs/superpowers/plans/fase-1-backend-core.md` — plan de Fase 1

## Qué falta para Fase 2 (Auth)

### Backend
1. **Router `/auth`** con endpoints:
   - `POST /auth/login` — valida credenciales, devuelve access_token en body + refresh_token en httpOnly cookie
   - `POST /auth/refresh` — renueva access_token usando refresh cookie
   - `GET /auth/me` — devuelve usuario autenticado
   - `POST /auth/logout` — invalida refresh_token
   - `POST /auth/register` — solo superuser puede crear usuarios
2. **Tabla `refresh_tokens`** — token_hash, user_id, expires_at
3. **Model `RefreshToken`** en SQLAlchemy
4. **Proteger todos los routers** con `get_current_user` (actualmente son públicos)

### Frontend
5. **Nueva página `/login`** — formulario de login
6. **AuthProvider** — maneja access_token en localStorage, refresh automático
7. **ApiProvider** — Axios instance con interceptors para Bearer + 401 refresh
8. **Protected routes** — wrapper que redirige a `/login` si no hay token
9. **Logout button** — en Header o menú de usuario

### Notas
- Los endpoints de negocio ya están creados pero SIN protección. La Fase 2 agrega auth middleware sin tocar la lógica de negocio.
- El frontend Next.js sigue en la raíz (no se movió a `frontend/` todavía).
- `dependencies.py` ya tiene `get_current_user` implementado pero NO se usa en ningún router todavía.
